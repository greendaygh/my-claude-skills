# 전문가 패널 역할 및 토론 프로토콜

## 패널 구성

| 역할 | 이름 | 페르소나 | 핵심 초점 |
|------|------|----------|----------|
| domain_expert | 도메인 전문가 | 20년+ 연구 경험의 선임 연구자 | 엔티티/관계 식별, 용어 정확성 |
| methodologist | 방법론 전문가 | 연구 설계/체계적 고찰 전문가 | 논문 품질, 재현성, 증거 수준 |
| statistician | 통계/데이터 전문가 | 생물통계학/데이터 분석 전문가 | 정량적 검증, 효과 크기, 일관성 |
| critical_reviewer | 비판적 검토자 | 10년+ 피어리뷰 경험의 학술 편집자 | 편향, 누락, 과잉 해석, 연구 갭 |

## 2사이클 × 2라운드 프로토콜

### 사이클 1: 초기 탐색

#### 라운드 1: 검색 전략 + 스키마 검토

**입력**: 도메인 설명 + 초기 스키마 JSON
**각 전문가 임무**:

| 전문가 | 라운드 1 임무 |
|--------|-------------|
| 도메인 전문가 | 핵심 검색 키워드, MeSH 용어, 주요 저널, 스키마의 도메인 적합성 |
| 방법론 전문가 | 포함/제외 기준, 연구 설계 필터, 메타분석 포함 여부 |
| 통계 전문가 | 표본 크기 기준, 효과 크기 보고 여부 필터, 정량적 속성 추가 제안 |
| 비판적 검토자 | 검색 편향 가능성, 누락 검색어, 과잉 스키마 항목 지적 |

**출력**: 합의된 검색 쿼리 + 수정된 스키마 v1.1

#### 라운드 2: 추출 검증 + 스키마 개선 권고

**입력**: 추출 결과 요약 + 스키마 외 발견 목록 + 현재 스키마
**각 전문가 임무**:

| 전문가 | 라운드 2 임무 |
|--------|-------------|
| 도메인 전문가 | 새로 추가해야 할 엔티티/관계 타입, 용어 정규화 오류 |
| 방법론 전문가 | 증거 수준 속성 보강, 재현성 의심 논문 플래깅 |
| 통계 전문가 | 정량적 속성 추가/수정 제안, 데이터 일관성 |
| 비판적 검토자 | 과잉 스키마 항목 제거, 검색 범위 조정 제안, 편향 지적 |

**출력**: 검증된 추출 결과 + schema_delta

### 사이클 2: 정제

#### 라운드 3: 정제된 전략 수립

**입력**: 정제된 스키마 v2.0 + 사이클 1 요약 + schema_delta
**각 전문가 임무**:

| 전문가 | 라운드 3 임무 |
|--------|-------------|
| 도메인 전문가 | 정제된 스키마의 도메인 적합성 최종 확인 |
| 방법론 전문가 | 보완 검색이 필요한 연구 설계 유형 |
| 통계 전문가 | 정량적 속성 정의의 적절성, 재추출 범위 |
| 비판적 검토자 | 사이클 1 편향 분석, 보완 검색 방향 제안 |

**출력**: 보완 검색 쿼리 + 재추출 범위 결정

#### 라운드 4: 최종 검증 + 합의

**입력**: 병합된 전체 추출 결과 + 스키마 v2.0 + 변경 이력
**각 전문가 최종 임무**:

| 전문가 | 라운드 4 임무 |
|--------|-------------|
| 도메인 전문가 | 이름 정규화 최종 확인, 관계 일관성 |
| 방법론 전문가 | 증거 수준 최종 평가, 재현성 우려 |
| 통계 전문가 | 데이터 일관성, 모순/중복 탐지 |
| 비판적 검토자 | 전체 편향 평가, 누락 핵심 관계, 과잉 추출 |

**출력**: 최종 검증 결과 + panel_confidence + 최종 보고서

## Task 호출 패턴

### 전문가 개별 호출

```
Task tool:
  subagent_type: general-purpose
  description: "{역할} 라운드 {N}"
  prompt: |
    당신은 {역할_설명} 전문가입니다.

    ## 배경
    도메인: {domain_description}

    ## 현재 단계
    사이클 {cycle_num}, 라운드 {round_num}: {round_title}

    ## 이전 맥락
    {previous_context - 슬라이딩 윈도우 직전 3개 발언}

    ## 검토 대상
    {round_specific_input}

    ## 임무
    {round_specific_tasks}

    ## 응답 형식
    간결하게 (3-5문장) 답변하세요.
    다른 전문가의 의견에 동의하거나 반박하세요.
    구체적인 근거를 제시하세요.
```

### 병렬 호출

각 라운드에서 4명을 **병렬로** 호출:

```
// 하나의 메시지에서 4개의 Task 호출
Task(도메인 전문가 라운드 N) ┐
Task(방법론 전문가 라운드 N) ├── 병렬 실행
Task(통계 전문가 라운드 N)  │
Task(비판적 검토자 라운드 N) ┘
         ↓
    Claude가 합의 종합
```

## schema_delta 형식

라운드 2에서 생성되는 스키마 변경 권고:

```json
{
  "add_entity_types": [
    {
      "label": "Pathway",
      "primary_key": "name",
      "properties": ["database", "pathway_id"],
      "reason": "10편 이상에서 경로 관련 엔티티 빈번히 발견"
    }
  ],
  "modify_entity_types": [
    {
      "label": "Gene",
      "add_properties": ["function", "pathway_membership"],
      "reason": "유전자 기능 정보가 관계 해석에 필수적"
    }
  ],
  "remove_entity_types": [],
  "add_relationship_types": [
    {
      "type": "PARTICIPATES_IN",
      "from_label": "Gene",
      "to_label": "Pathway",
      "properties": ["role"],
      "reason": "유전자-경로 관계가 빈번하게 추출됨"
    }
  ],
  "modify_relationship_types": [],
  "remove_relationship_types": [],
  "additional_search_queries": [
    "signaling pathway CRISPR"
  ],
  "extraction_notes": [
    "Methods 섹션에서 실험 조건을 더 체계적으로 추출할 것",
    "Table 데이터에서 정량적 속성을 우선 추출할 것"
  ]
}
```

## 합의 프로토콜

1. **독립 의견 수집**: 4명 병렬 → 각자 독립 답변
2. **동의율 계산**: 각 항목에 대해 동의/반대 비율
3. **합의 기준**: `consensus_threshold: 0.75` (4명 중 3명 이상 동의)
4. **불일치 처리**:
   - 75% 미만 동의: Claude가 양측 의견 요약 후 2차 발언 요청 (추가 Task 호출)
   - 2차 후에도 불일치: 다수결 채택 + 불일치 기록
5. **최종 합의 문서**: 각 결정 항목 + 동의율 + 반대 의견 요약

## 사이클 간 연속성

- 사이클 2의 패널에게 사이클 1의 합의 요약을 컨텍스트로 전달
- 같은 전문가 페르소나 유지
- 사이클 1의 주요 결정과 그 근거를 명시적으로 참조
- schema_delta 적용 결과를 사이클 2 패널에 공유

## 모니터링 모드 간소화 패널

신규 논문 추가 시 간소화된 1라운드 패널:

```
- 4명 전문가 병렬 호출 (1라운드)
- 입력: 새 추출 결과 + 기존 그래프 요약
- 임무: 새 추출의 기존 그래프 일관성 확인
- 신뢰도 점수 부여
- 불일치 시에만 2차 발언
```
